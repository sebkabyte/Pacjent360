Niezależny prototyp open source

Mapa zdarzeń pacjenta. Film z życia, nie folder z dokumentami.

Pacjent 360 porządkuje dokumenty, wizyty, leki, objawy, obserwacje i ustalenia w jedną chronologiczną mapę. Pacjent, opiekun i lekarz widzą tę samą historię, ale pracują w widokach dopasowanych do swojej roli.

Problem

System widzi dokumenty. Człowiek potrzebuje ciągłości.

Historia pacjenta żyje w wynikach, wypisach, receptach, notatkach, pamięci rodziny i krótkich rozmowach podczas wizyt. Same pliki nie pokazują przebiegu zdarzeń, zmian stanu, rozbieżności ani tego, co trzeba wyjaśnić przed kolejnym kontaktem z lekarzem.

Pacjent 360 buduje warstwę kontekstu: źródła, zdarzenia, pytania, braki danych, zgody i zadania organizacyjne. Celem jest lepsza rozmowa medyczna, nie automatyczna decyzja.

Produkt

Dwa kokpity i desktop lekarski. Pełne dane, różny rytm pracy.

Pacjent i opiekun potrzebują jasnych następnych kroków. Lekarz potrzebuje szybkiego kontekstu, źródeł, rozbieżności i pytań DITL. Wspólną warstwą jest ta sama mapa zdarzeń.

Bezpieczne granice

System porządkuje kontekst. Decyzja zostaje po stronie lekarza.

Pacjent 360 nie jest wyrobem medycznym, nie jest systemem ustalania kolejności obsługi, nie diagnozuje i nie tworzy zaleceń terapeutycznych.

System może

zbierać źródła, układać zdarzenia, pokazywać braki i zadania organizacyjne.

System pyta

czy lista leków, objaw, dokument lub obserwacja wymagają potwierdzenia.

Lekarz decyduje

potwierdza, odrzuca, edytuje i podejmuje decyzje kliniczne.

System nie robi

nie zastępuje lekarza, nie jest usługą CeZ/NFZ/IKP i nie obchodzi systemów publicznych.

Status alpha

To publiczny prototyp do walidacji, nie gotowy system kliniczny.

Demo działa wyłącznie na fikcyjnych lub syntetycznych danych. Celem jest sprawdzenie, czy format mapy zdarzeń, raportu kontekstowego i pytań DITL jest zrozumiały oraz przydatny przed rozmową z lekarzem.

Dane fikcyjne

Nie używamy realnych danych pacjentów w publicznym demo ani w zgłoszeniach issue.

Jasne granice

Nie CDSS, nie wyrób medyczny, nie system triażu, nie dokumentacja medyczna.

Review przed funkcją

Każda nowa automatyzacja ma przejść clinical safety, privacy, security i UX review.

Możliwości rozwoju

Od prototypu alpha do warstwy kontekstu pacjenta.

Najpierw walidujemy mapę, raport i workflow. Następnie rozwijamy aplikację pacjenta, opiekuna, desktop lekarski, zgody, interoperacyjność i asystentów operacyjnych bez autonomii klinicznej.
